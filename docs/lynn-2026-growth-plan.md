### 1.2 当前阶段定位

- ✅ 能做跨组件（TiDB / TiKV / PD / Dashboard / Cloud）的系统级改动

### 1.3 核心风险识别

> **最大的风险不是能力不够，而是"价值被稀释、被误读、被吞没"。**

具体表现：

| 问题 | 表现 | 影响 |

你说的提升空间是"希望对新事物有更深入了解"——这是**表面问题**。

**真实问题是**：你在执行层面已经是顶级，但缺乏**杠杆效应**。

- 时间换产出：172个issue、500+PR，靠的是时间投入，不是杠杆

## 三、2026 年详细年度计划

### 3.1 年度总目标

> 到 2026 年底，被清晰识别为：
> 「**TiDB 诊断与可观测性方向的系统级负责人 / 核心架构者之一**」

**衡量标准**：
- 不在场时，别人讨论"慢 SQL / TopSQL / Clinic / 诊断闭环"，会自然提到你
- 新人或跨团队会说："这个得找 Lynn，她最懂这块"

**能力主题**：

> "让 TiDB 的性能问题，可以被'**解释**'而不是'**猜**'"

所有产出都要能回答这 3 个问题之一：
1. 为什么慢了？
2. 什么时候开始变的？
3. 是不是异常？还是预期变化？

#### 交付1：诊断闭环模型 (TiDB Performance Diagnosis Loop - TPDL)

**五步模型框架**：

| 步骤 | 名称 | 核心问题 | 工具映射 |
|-----|------|---------|---------|
| 1 | **Trigger (触发)** | 什么情况下应该开始诊断？ | Alert Rules、Multi-dim Slow Log |
| 2 | **Capture (捕获)** | 捕获什么信息？ | TopSQL、Slow Log、Metrics |
| 3 | **Compare (对比)** | 和什么对比？ | Clinic 对比功能 |
| 4 | **Explain (解释)** | 为什么变了？ | Clinic 解释功能 |
| 5 | **Act (行动)** | 应该做什么？ | 内置建议 |

**交付要求**：
- 写进设计文档
- 写进 PR description
- 体现在 Clinic/Slow Query 页面交互逻辑

> 名字 = 认知锚点。没有名字的东西 = 不存在。

#### 交付2：Clinic Slow Query 标杆功能

**核心要求**：能回答问题，而不只是展示数据。

必须支持的诊断场景：
- "这次变慢，是 SQL 行为变了？QPS 变了？Retry 变了？Scan 变了？"
- "这是结构性退化，还是负载变化？"

> 表格、Δ%、红绿箭头只是形式。重点是：定义"什么变化值得人注意"。

#### 交付3：深度技术文章

**候选主题**（选1个）：

| 选项 | 标题 | 类型 | 推荐度 |
|-----|------|------|-------|
| A | 《从 Slow Log 到诊断闭环：TiDB 可观测性设计的 3 个认知转变》 | 设计思考 | ⭐⭐⭐⭐⭐ 首选 |
| B | 《TopSQL 源码解析：如何在不影响性能的情况下捕获 SQL 资源消耗》 | 技术深度 | ⭐⭐⭐⭐ |
| C | 《TiDB 性能诊断实战：从"感觉慢了"到"知道为什么慢"》 | 实践指南 | ⭐⭐⭐⭐ |

**选项 A 结构建议**：
```
1. 为什么"看日志"解决不了性能问题？
2. 诊断的本质是"对比"而非"观察"
3. TiDB 诊断闭环模型 (TPDL) 的设计
4. 实际案例：一个慢查询的诊断过程
```

## 附录：诊断闭环模型 (TPDL) 模板

```markdown
# TiDB Performance Diagnosis Loop (TPDL)

## 概述

TPDL 是 TiDB 性能诊断的标准方法论，包含五个步骤：
Trigger → Capture → Compare → Explain → Act

## 五步详解

### 1. Trigger (触发)

**核心问题**：什么情况下应该开始诊断？

**触发条件**：
- 延迟突增（P99 > 阈值）
- QPS 下降
- 资源异常（CPU/Memory/IO）
- 用户反馈

**工具支持**：
- Alert Rules
- Multi-dimensional Slow Log 触发规则

### 2. Capture (捕获)

**核心问题**：捕获什么信息？

**捕获内容**：
- SQL 文本 + Digest
- 执行计划
- 资源消耗（CPU/内存/IO）
- 上下文信息（时间、负载、配置）

**工具支持**：
- TopSQL
- Slow Log
- Metrics

**关键原则**：完整上下文，不是碎片

### 3. Compare (对比)

**核心问题**：和什么对比？

**对比维度**：
- 时间维度：和历史基线对比（昨天、上周）
- 空间维度：和其他 SQL/节点对比

**工具支持**：
- Clinic 对比功能

### 4. Explain (解释)

**核心问题**：为什么变了？

**归因分类**：
- SQL 变化：SQL 文本或执行计划变了
- 数据变化：数据量或分布变了
- 负载变化：QPS 或并发变了
- 系统变化：配置或版本变了

**工具支持**：
- Clinic 解释功能

### 5. Act (行动)

**核心问题**：应该做什么？

**行动类型**：
- 优化 SQL
- 添加索引
- 调整配置
- 扩容资源
- 升级版本

**工具支持**：
- 内置建议
- 文档链接

## 工具映射表

| 步骤 | 现有工具 | 2026 增强 |
|-----|---------|----------|
| Trigger | Alert Rules | Multi-dim Slow Log |
| Capture | TopSQL, Slow Log | TopSQL topRU |
| Compare | 手动 | Clinic 对比功能 |
| Explain | 手动分析 | Clinic 解释功能 |
| Act | 文档 | 内置建议 |
```
